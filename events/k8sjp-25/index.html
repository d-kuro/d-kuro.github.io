入門、Kubernetes Persistent Volume

  • ストレージ種別
    • ブロックストレージ
      • Raw デバイス
      • ファイルシステムは自由に選択
    • ファイルストレージ
      • ネットワークドライブ
      • ファイルシステムはストレージ任せ
    • オブジェクトストレージ
      • オブジェクト単位でアクセス
      • ファイルシステムに依存しない
  • Persistent とは
    • Ephemeral
      • Pod が削除されるとデータが消える
      • emptyDir
    • Persistent
      • Pod が削除されるとデータが消えない
      • Persistent Volume
  • Kubernetes v1.9 で CSI がサポートされた
    • v1.13 で CSI が GA
  • ストレージのモデル
    • PersistentVolumeClaim
    • PersistentVolume
    • StorageClass
  • PersistentVolume, StorageClass は全 namespace で共通
    • クラスタ全体の管理者権限が必要
  • PersistentVolumeClaim は namespace 単位
  • PersistentVolumeClaim (PVC)
    • Volume の要求仕様
      • strage type
      • volume size
      • access mode
  • ブロックストレージ
    • LBA のアクセスのためロックのメカニズムを持っていない
    • 1台のホストから write されるようにマウント
  • ファイルストレージ
    • ファイル単位でロックできる
    • Open 時に取得したロックを他のホストにも伝搬
    • 複数ホストから R/W でマウントできる
  • PersistentVolume (PV)
    • Volume 表す
  • Reclaim Policy
    • Retain
      • PVC 削除時も消さない
    • Recycle (deprecated)
      • Volume は残ったままで中のデータを消す
    • Delete
      • PVC 削除と同時に PV を消す
  • StorageClass
    • ストレージプールを表す
    • Procisioner がストレージ装置の API を呼び出す
  • StatefulSet
    • PVC, PV を便利かつ安全に使う
    • Pod に連番を付与
    • Pod を更新する場合必ず前の番号の Pod の処理が終了してから実行
    • Pod の生成時に PVC を自動で生成することもできる
    • StatefulSet 消しても Volume は消えない
  • Rolling Update
    • Pod は再作成されても Volume は再作成されない
    • 一回デタッチした後に再度アタッチされる

KubernetesにおけるCSIについて

  • ボリュームプラグインの歴史
    • v1.8 まで Kubernetes のリポジトリにしかなかった
    • v1.9 から CSI の対応が Alpha でリリース
  • CSI 移行のモチベーション
    • ストレージプロバイダは CSI を実装するだけでいい
    • ボリュームプラグインのコードを Kubernetes から除外できる
    • リリースフローを別にすることができる
  • CSI
    • CSI ドライバの通信方法
    • CSI ドライバが提供する機能
    • gRPC の protobuf
  • CSI ドライバの通信方法
    • UNIX ドメインソケットで通信
  • Kuberntes と CSI の連携
    • Sidecar
    • Node 側は kubelet に CSI プラグインを読み込ませる
  • CSIDriver Object
    • 必須じゃない
    • CSI Driver の設定を変えることができる
  • StorageClass Object
    • プラグインの指定
    • CSI ドライバのデフォルトの設定を行うことができる
