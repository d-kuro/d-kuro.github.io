minne の開発環境の変革とこれから

  • Kubernetes でのローカル開発環境
    • telepresence で起動
    • リモートのアクセスをローカルの Docker に流すことができる
    • ローカルで起動するコンテナが 1 つになった
    • 個人の環境への依存度が下がった
  • リモートの開発環境
    • 個人で 1 Pod を所有できるようにした
    • Nginx Ingress Contoller でアクセスを流す
  • Skaffold
    • image の作成や deploy の自動化
    • yaml で管理できる
  • kustomize
    • 環境ごとの差分をいい感じに吸収して manifest を生成できる
  • kaniko
    • コンテナ内で image build をしてくれる
    • Docker command が不要
    • 指定したレジストリに push してくれる
  • External DNS
    • Ingress / Service の設定をもとに DNS レコードの更新をしてくれる
  • telepresence
    • リモートのクラスタに接続しながらローカルで実行したサービスにトラフィックを流すことができる

Kubernetes 1.16 アルファ機能を先取り！エフェメラルコンテナ

  • デバッグに使える kubectl のサブコマンド
    • kubectl run
      • Pod の作成
    • kubectl logs
      • ログの表示
    • kubectl exec
      • 任意のコマンド実行
    • kubectl port-forward
      • コンテナのポートをローカルに転送する
    • kubectl cp
      • コンテナとローカル間のファイル転送
      • 将来的に削除されそう
    • kubectl top
      • Pod や Node のリソース状況を表示
  • kubectl exec の課題
    • コンテナ内で任意のコマンドを実行する
    • コンテナに含まれているコマンドしか実行できない
    • 開発者は scratch や distroless をベースイメージに使いたい
    • distroless
      • Google が公開するランタイムのみを含むコンテナイメージ
    • 最小イメージは運用の負担が少ない
      • 余計なものの脆弱性とかで悩むことが減る
    • アタックサーフェスが狭い
    • サイズが小さいイメージはダウンロードから実行までの時間が最小化される
    • ディスクの使用量も少なくて済む
    • 実行中のコンテナのトラブルシューティングが困難
    • デバッグ用のサイドカーをすべての Pod に仕込むのはリソースの無駄
    • 最低限のユーティリティを仕込む場合もデバッグする人がイメージを作成した人とは限らない
      • 他社が用意したイメージに手を入れるのは面倒
  • エフェメラルコンテナ
    • 1.16 からの新機能
    • 実行中の Pod にデバッグを目的としたコンテナを後から追加する
    • まだアルファ
    • パブリッククラウドでは使えない
    • Pod.spec.sharePorcessNamespace を有効にする必要がある
  • ユースケース
    • オペレーション
    • デバッグ
    • 自動化
      • 全ての Pod で何かスクリプトを実行したいみたいなケース
    • テクニカルサポート
  • 制約
    • 実行に必要なリソースを確保する保証がない
    • 再起動されない
    • 通常のコンテナで使用できる多くのフィールドが使えない
      • ports
      • resources
      • probe
    • なにかの用途で便利に使われないようにするための制約
  • 使い方
    • kubectl にはまだ実装されていない
    • プラグインを使う
    • debug コマンド提供後も exec は提供される
  • API
    • Pod のフィールド追加
    • ephemeralContainers
    • ephemeralContainerStatuses
  • API Server
    • ephemeralContainers サブリソースの追加
    • サブリソース単位で RBAC できる
    • Pod スペックの定義による作成時の設定を許可しない
    • Pod が実行された後じゃないとフィールドが更新できないようになっている
    • 内部的な kind: EphemeralContainers で更新を行う
  • kubelet
    • Pod を watch して spec.ephemeralContainers に変更があればエフェメラルコンテナを起動する
    • 起動は CRI を通して行われる
  • kubectl
    • サブコマンドの追加
    • ずっとプラグインの可能性もなくもない
  • マイルストーン
    • kubectl コマンドの実装 (v1.17)
    • container namespace targeting を kubectl がサポート (v1.17)
    • アルファステージでのフィーチャーコンプリート (v1.18)
